# Neo MCP — Web Connector Implementation

## Overview

Neo MCP supports OAuth 2.0 so users can connect directly from **Claude.ai** and **ChatGPT** web
interfaces without installing anything. The OAuth layer is a thin wrapper — the "access token"
issued is the user's `NEO_SECRET_KEY` itself. Neo's backend already validates Bearer tokens, so
nothing changes on the backend side.

---

## What's implemented

### OAuth 2.0 Authorization Server (`src/neo_mcp/oauth.py`)

| Endpoint | Method | Purpose | Status |
|----------|--------|---------|--------|
| `/.well-known/oauth-protected-resource` | GET | RFC 9728 — points to our auth server | ✅ |
| `/.well-known/oauth-authorization-server` | GET | RFC 8414 — lists OAuth endpoints | ✅ |
| `/oauth/authorize` | GET | Renders "Enter your Neo secret key" HTML form | ✅ |
| `/oauth/authorize` | POST | Validates key, issues auth code, redirects back | ✅ |
| `/oauth/token` | POST | PKCE S256 exchange — returns secret key as `access_token` | ✅ |
| `/oauth/revoke` | POST | Token revocation no-op (tokens are stateless) | ✅ |

**Security:**
- PKCE S256 enforced — codes cannot be intercepted without `code_verifier`
- Auth codes expire after 10 minutes
- Single-use codes (deleted on exchange)
- `redirect_uri` restricted to `https://` or `http://localhost` — no open redirect
- Secret key never logged or stored permanently

### Bearer-only auth on `/mcp` (`src/neo_mcp/server.py`)

Previously required both `x-access-key` and `Authorization: Bearer`. Now accepts Bearer token
alone — OAuth connectors (Claude.ai, ChatGPT) only send this header.

On missing auth, returns:
```
HTTP 401
WWW-Authenticate: Bearer resource_metadata="https://mcp.heyneo.so/.well-known/oauth-protected-resource"
```
This `WWW-Authenticate` header triggers automatic OAuth discovery in compliant clients.

### Cloud deployment mode (`src/neo_mcp/server.py`)

Web connector users have no local VS Code/Cursor daemon. Task submission auto-detects the right
mode via `_resolve_deployment(deployment_id)`:

| Transport | `deployment_id` available? | `deployment_type` sent | Workspace prefix |
|-----------|---------------------------|------------------------|-----------------|
| `stdio` | any | `vscode` | ✅ included |
| `http` | yes (env var pinned) | `vscode` | ✅ included |
| `http` | no (web connector) | `cloud` | ❌ omitted |

Override: `NEO_DEPLOYMENT_TYPE=cloud` or `NEO_DEPLOYMENT_TYPE=vscode` env var.

### `neo_get_files` tool

Downloads all files generated by a completed task:

1. `POST /v1/export-artifacts?thread_id=...&upload_mode=multi` — triggers S3 upload
2. Polls export job status until complete
3. `GET /v2/thread/{thread_id}/files` — fetches presigned S3 download URLs
4. Fetches each file and returns contents inline as fenced code blocks

**Current status: ⚠️ partially blocked** — see Known Issues below.

## OAuth flow (end-to-end)

```
Claude.ai / ChatGPT
  │
  ├── User adds connector URL: https://mcp.heyneo.so/mcp
  │
  ├── GET /.well-known/oauth-protected-resource
  │         ← { authorization_servers: ["https://mcp.heyneo.so"] }
  │
  ├── GET /.well-known/oauth-authorization-server
  │         ← { authorization_endpoint, token_endpoint, ... }
  │
  ├── Redirect → /oauth/authorize?client_id=...&code_challenge=...&state=...
  │         User sees: "Enter your Neo secret key"
  │         User types: sk-v1-...
  │         ← Redirect back with ?code=<random>&state=...
  │
  ├── POST /oauth/token { code, code_verifier, grant_type }
  │         ← { access_token: "sk-v1-...", token_type: "bearer", expires_in: 7776000 }
  │
  └── POST /mcp  (Authorization: Bearer sk-v1-...)
            ← MCP tools available
```

## Known Issues

### `neo_get_files` — export requires JWT auth

`POST /v1/export-artifacts` returns `{"detail": "Invalid token"}` when called with
`Authorization: Bearer sk-v1-...`. This endpoint requires a JWT token (browser OAuth login),
not an API key.

**Impact:** `neo_get_files` cannot trigger the S3 export step with the current auth.
The `/v2/thread/{thread_id}/files` endpoint (which returns presigned URLs) works fine with
the API key, but it returns `"No artifact found"` until after an export is triggered.

**Options under consideration:**
- Find an API-key-compatible way to trigger export
- Auto-export on task completion server-side
- Return file contents via the message stream instead (ask Neo to print files inline)

## Quick local test

```bash
# Start HTTP server
NEO_TRANSPORT=http NEO_SECRET_KEY=sk-v1-... NEO_PUBLIC_URL=http://localhost:8000 \
  python3 src/neo_mcp/server.py

# Verify discovery
curl http://localhost:8000/.well-known/oauth-protected-resource | jq
curl http://localhost:8000/.well-known/oauth-authorization-server | jq

# Open authorize page in browser
open "http://localhost:8000/oauth/authorize?response_type=code&client_id=test&redirect_uri=http://localhost:9999/cb&code_challenge=VmZPH2sl5AAFM0zCOaICG9Ttg3flvh91ckpB3e8TVRU&code_challenge_method=S256&state=xyz"

# Exchange code for token (after form submit)
curl -X POST http://localhost:8000/oauth/token \
  -d "grant_type=authorization_code&code=<code>&code_verifier=test_verifier_abc123456789012345678901234567890"
```
